Chapter 2. Tokens & embeddings: how text becomes numbers

~35 min read · Lab: labs/02-bpe · Builds on: Ch 1 (vectors, dot products)

Models cannot read. Before any intelligence happens, text must become integers (tokenization), and integers must become vectors (embedding). These two "boring preprocessing steps" explain LLM pricing, several famous failure modes, why context windows are measured the way they are, and — once we reach RAG — why retrieval works at all. Engineers who skip this chapter pay for it later, literally: token counting is the unit of every LLM invoice.

2.1 Byte-Pair Encoding, built by hand

BPE (Byte-Pair Encoding) builds a vocabulary by starting from single characters (or bytes) and repeatedly merging the most frequent adjacent pair into a new token, for a fixed number of merges. The learned merge list IS the tokenizer.
Corpus (with word frequencies): low ×5, lower ×2, newest ×6, widest ×3. Represent each word as characters + end marker _:
l o w _ ×5 · l o w e r _ ×2 · n e w e s t _ ×6 · w i d e s t _ ×3

Count adjacent pairs: (e,s) appears 6+3 = 9 (in newest, widest) — the winner. Merge 1: e+s → es
Now n e w es t _ ×6, w i d es t _ ×3. Top pair: (es,t) = 9. Merge 2: es+t → est
Then (est,_) = 9 → Merge 3: est+_ → est_. Then (l,o) = 5+2 = 7 → Merge 4: l+o → lo. Then (lo,w) = 7 → Merge 5: lo+w → low.

After 5 merges the vocabulary contains est_, low as single tokens. "lowest" — a word never seen in training — tokenizes to low + est_: two meaningful units. That is the entire genius of subword tokenization: common patterns compress, rare words decompose, nothing is ever "unknown."

Production tokenizers (GPT-4's o200k_base, Llama's SentencePiece) are this exact algorithm run over terabytes with 100k–256k merges, operating on bytes rather than characters so that any UTF-8 string — emoji, Hindi, code — tokenizes without an unknown-token escape hatch.

2.2 From token IDs to meaning: the embedding matrix

A token ID is just an index. The model's first layer is an embedding matrix E of shape [vocab_size × d_model] — for GPT-3: 50,257 × 12,288. Token 464 → row 464: a 12,288-dimensional vector. These rows are learned parameters, tuned by the same gradient descent as everything else, and they arrange themselves so that tokens used similarly end up geometrically close — because that arrangement reduces next-token-prediction loss.

The embedding matrix is a learned dictionary where definitions are locations. Nobody told the model "king relates to queen as man to woman" — but arranging royalty along one direction and gender along another was the arrangement that best predicted text, so gradient descent found it. The famous king − man + woman ≈ queen is real linear structure in trained embedding spaces (clearest in word2vec-era static embeddings).

Static vs. contextual — the upgrade that changed everything

Static (word2vec, 2013)Contextual (transformers, 2018+)
Vectors per wordExactly one, foreverA new vector every time, computed from surrounding context
"bank" in "river bank" vs "bank account"Same vector — ambiguity unresolvedDifferent vectors — attention pulled in "river" vs "account"
How obtainedShallow prediction task (skip-gram: predict neighbors)Full transformer forward pass; sentence embeddings pool the output
Role todayHistorical + intuition builderEverything: the E matrix inside LLMs, and the sentence-embedding models powering RAG

The embedding models you'll use for RAG (OpenAI text-embedding-3, Cohere embed, open-source bge/e5) are small transformers trained with contrastive objectives: pull (query, relevant passage) pairs together, push unrelated pairs apart. That training is why "money back" and "refund policy" land near each other — the property RAG's entire retrieval step depends on. Model quality is tracked on the MTEB leaderboard; dimension typically ranges 384–3072, and Matryoshka-trained models let you truncate vectors (3072 → 256) for cheaper storage with modest quality loss — an increasingly common cost lever.

"LLM token embeddings and RAG sentence embeddings are the same thing." — Related but distinct. The E matrix maps single tokens for the LLM's internal use. RAG uses separate sentence-embedding models whose entire output is one vector per passage, contrastively trained specifically so cosine similarity means "these texts are relevant to each other." Using an LLM's internal token embeddings for retrieval works badly; interviewers occasionally probe exactly this confusion.

BPE in 30 lines

TypeScript — training loop only (full tokenizer with encode/decode in the lab)

function trainBPE(words: Map<string[], number>, numMerges: number): [string, string][] {
  const merges: [string, string][] = [];
  for (let i = 0; i < numMerges; i++) {
    const pairCounts = new Map<string, number>();
    for (const [symbols, freq] of words)         // count adjacent pairs, weighted by word frequency
      for (let j = 0; j < symbols.length - 1; j++) {
        const key = symbols[j] + "" + symbols[j + 1];
        pairCounts.set(key, (pairCounts.get(key) ?? 0) + freq);
      }
    const best = [...pairCounts.entries()].sort((a, b) => b[1] - a[1])[0];
    if (!best) break;
    const [a, b] = best[0].split("");
    merges.push([a, b]);
    for (const [symbols] of words) {              // apply the merge everywhere
      for (let j = 0; j < symbols.length - 1; j++)
        if (symbols[j] === a && symbols[j + 1] === b)
          symbols.splice(j, 2, a + b);
    }
  }
  return merges;                                  // the merge list IS the tokenizer
}

Interview ammunition

Why subword tokenization instead of word-level or character-level?

Word-level: unbounded vocabulary, out-of-vocabulary failures, no sharing between "run/running/runner." Character-level: tiny vocabulary but sequences 4–5× longer — and attention cost grows quadratically with sequence length, so that's a compute disaster. Subword (BPE) is the compromise: frequent strings become single tokens (compression), rare words decompose into meaningful pieces (coverage), vocabulary stays fixed (~50–250k). Byte-level BPE additionally guarantees no input is ever unrepresentable.

Why do LLMs fail at counting letters in a word?

The model receives token IDs, not characters. "strawberry" arrives as one or a few opaque integers; the letters inside are not part of the representation. Character-level information exists only to the degree training data happened to spell words out. Correct fix in production: route such tasks to code execution, not to the model's intuition.

Your prompt is identical but token counts differ across model providers. Why?

Different tokenizers: different vocabularies, merge orders, and byte-handling. The same string can be 100 tokens in one and 130 in another — affecting cost, latency, and context fit. This is also why "token count" must be computed with the target model's own tokenizer, never estimated by characters/4 in anything billing-critical.

How does an embedding model differ from an LLM if both are transformers?

Objective and output. LLM: causal next-token prediction, outputs a distribution over the vocabulary, autoregressive. Embedding model: contrastive training on (query, passage) pairs, outputs one pooled vector, bidirectional attention (no causal mask — it can see the whole input). Same architecture family, completely different training targets and uses.

When would you truncate embedding dimensions, and what's the trade-off?

With Matryoshka-trained models, the first k dimensions carry most information, so truncating 3072→512 cuts vector storage and search cost ~6× with a small recall drop. Worth it when the corpus is millions of chunks and storage/latency dominate; not worth it for small corpora where quality is the bottleneck. Quantify with recall@k on your own eval set before committing.

Whiteboard summary

BPE: count adjacent pairs → merge most frequent → repeat k times; merge list = tokenizer; unseen words decompose (lowest = low + est_).
Bytes not chars → nothing unrepresentable. ~4 chars/token English; more elsewhere.
E matrix [vocab × d]: token ID → learned row vector; similar usage → nearby vectors.
Static = one vector/word (word2vec); contextual = computed per occurrence (transformers).
RAG embeddings: separate contrastively-trained models; cosine ≈ "relevant to each other."
