Chapter 12. RepoSage — the flagship build

~45 min read · The capstone that uses every prior chapter

Now you assemble everything into one portfolio-defining system. RepoSage is a self-correcting, multi-agent codebase intelligence engine: point it at any Git repository and it answers not just "where is X?" but "why is X built this way?" — by fusing three retrieval modalities (code, git history, docs including diagrams), routing across specialist agents, and grading its own answers before they ship. This chapter gives you the architecture, the component contracts, the interview framing, and the milestone-mapped build plan.

Imagine the one engineer who's been on a team for ten years — ask them anything and they just know, because they've read all the code, lived through every refactor, and remember the discussions behind each decision. RepoSage is that engineer as software. The "why" questions are the differentiator: answering them requires joining current code with git archaeology and design docs — three modalities no chat-with-your-PDF demo touches.

12.1 Why this is a flagship, not a tutorial clone

  • Nobody's demo answers "why." "Why two HTTP clients?" needs code + the commit that introduced the second + the PR rationale — a systems story, not a retrieval toy.
  • Self-correction is visible. The UI shows the critic rejecting a draft and the router retrying — reviewers watch the advanced architecture work.
  • Demoable live on any public repo. An interviewer names a repo; it works. That's a different level of credibility than screenshots.
  • Every chapter is load-bearing: tokenization (cost budgets), attention (context ordering), embeddings/ANN (retrieval), hybrid + Graph RAG (the three indexes), agents (orchestration), evals (the golden set), production (caching, injection defense). You can defend any layer because you built it.

12.4 One question, end to end

You ask: "Why does this project use two different HTTP clients?"

1 · Router (cheap model, ~1s): needs code agent ("find HTTP client usages") + history agent ("commits introducing HTTP clients").
2 · Code agent: hybrid search finds an axios wrapper in lib/http.ts and raw fetch in edge/; the call graph shows which modules use which. History agent: finds commit "switch edge functions to fetch — axios breaks on Cloudflare Workers," 2024.
3 · Synthesizer (strong model): "Two clients are deliberate: axios on the Node server for interceptor-based auth [1]; fetch in edge functions because axios is incompatible with the Workers runtime — see commit abc123 [2]."
4 · Critic: checks each claim against the chunks. The interceptor claim has no supporting chunk → grade fails → re-search "axios interceptor" → found in lib/http.ts:15 → re-synthesize → grade passes.
5 · UI: answer streams with clickable citations; the trace panel shows all five steps, timings, and ~$0.02 cost.

12.6 Presenting it in interviews

The build is half the value; framing it is the other half. Prepare these:

  • The 60-second pitch: problem (onboarding to a codebase is slow; "why" knowledge is tribal) → approach (fuse code + history + docs, self-correct, cite) → outcome (live demo on any repo + your eval numbers).
  • One hard number per layer: "hybrid + rerank raised context precision from 0.61 to 0.89 on my golden set"; "prompt caching cut cost 84%"; "the critic loop caught 30% of drafts as ungrounded." Numbers from your evals are what make it real.
  • A failure story: the prompt-injection you wrote and patched, or a retrieval failure you diagnosed via tracing and fixed. Senior signal = you talk about what broke and how you found it.
  • Trade-offs you chose: "I used pgvector not Pinecone because…", "I used a workflow not a fully autonomous agent because errors compound." Owning trade-offs beats claiming everything is optimal.

Whiteboard summary — the whole track, one system

RepoSage = router → {code, history, docs agents} → synthesizer → critic (bounded self-correct), over 3 indexes (vector+BM25, knowledge graph, multimodal), through one gateway, gated by a golden eval set.
Differentiator: answers "why" by fusing code + git + docs, and shows its self-correction.
