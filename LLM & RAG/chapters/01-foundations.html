Chapter 1. The math that runs everything

~40 min read · Lab: labs/01-math · Prerequisite: comfortable with any programming language

Every impressive thing an LLM does reduces to four mathematical operations repeated billions of times: dot products (measuring similarity), matrix multiplications (transforming representations), softmax (turning scores into probabilities), and gradient descent (learning from mistakes). Master these four with real intuition — not just formulas — and nothing in the next eleven chapters will feel like magic. This chapter builds each one from zero, then trains a tiny neural network by hand so you feel backpropagation in your fingers.

1.1 Vectors: meaning as position

A vector is an ordered list of numbers, interpreted as a point (or arrow) in space. In LLMs, every token, every intermediate representation, and every "thought" is a vector — typically 768 to 16,384 numbers long.

Why lists of numbers? Because similarity becomes geometry. If "dog" and "puppy" are both points in a 1,536-dimensional space, and the training process pushed related concepts near each other, then "how similar are these words?" becomes "how close are these points?" — a question with an exact, fast, computable answer.

The dot product — the single most executed operation in AI

a · b = a₁b₁ + a₂b₂ + … + aₙbₙ

Multiply matching positions, add everything up, get one number. Geometrically: a · b = |a| |b| cos θ — the product of the lengths times the cosine of the angle between them. Large positive → pointing the same way. Zero → perpendicular (unrelated). Negative → opposing.

Let a = [2, 1, 0] and b = [1, 3, 0].
a · b = (2)(1) + (1)(3) + (0)(0) = 5.
|a| = √(4+1+0) = √5 ≈ 2.236    |b| = √(1+9+0) = √10 ≈ 3.162
cos θ = 5 / (2.236 × 3.162) = 5 / 7.071 ≈ 0.707 → θ = 45°.
The vectors are moderately aligned — and you just computed your first similarity score.

Cosine similarity — normalizing away magnitude

cos(a, b) = (a · b) / (|a| |b|)    ∈ [−1, 1]

Why divide by the lengths? Because in embedding spaces, direction encodes meaning; length often encodes artifacts (word frequency, sentence length). A 2,000-word document about dogs and a 20-word note about dogs should be "similar" — cosine makes them so; raw dot product would let the longer document dominate.

"Cosine similarity and dot product are different algorithms with different results." — Not when vectors are normalized to length 1 (as most embedding APIs do, including OpenAI's). Then |a| = |b| = 1, so cosine = dot product exactly. This is why vector databases often default to dot product: it's cosine, minus two divisions. In an interview, saying "we normalize at index time so dot product IS cosine" signals you actually understand the geometry.

1.2 Matrices: machines that transform meaning

A matrix is a grid of numbers. Multiplying a vector by a matrix produces a new vector: each output element is a dot product between the input and one row of the matrix. A matrix is therefore a bundle of learned "questions" — each row asks the input one dot-product question, and the output vector is the list of answers.

This reframing matters enormously for the transformer chapter: when you see Q = W_q · x, read it as "the matrix Wq asks the token vector x a battery of questions, producing its Query vector." A neural network layer is exactly output = activation(W · input + b) — one transformation, one nudge (bias), one nonlinearity.

Think of a matrix as a machine that moves every point in space simultaneously — rotating, stretching, projecting. Deep learning = stacking many such machines and letting gradient descent tune what each one does. "Deep" just means "many transformations composed."

1.3 Softmax: from scores to probabilities

After all its layers, an LLM produces one raw score (a logit) per vocabulary token — say 100,000 numbers like [2.1, −0.3, 5.7, …]. To pick the next token we need probabilities: non-negative, summing to 1. Softmax does this:

softmax(zᵢ) = e^(zᵢ) / Σⱼ e^(zⱼ)
Logits for three candidate tokens: z = [2, 1, 0].
e² = 7.389, e¹ = 2.718, e⁰ = 1.000 → sum = 11.107
P = [7.389/11.107, 2.718/11.107, 1/11.107] = [0.665, 0.245, 0.090]

Now apply temperature T = 2 (divide logits by T before softmax): z/T = [1, 0.5, 0]
e¹=2.718, e^0.5=1.649, e⁰=1 → sum = 5.367 → P = [0.506, 0.307, 0.186]
Same ranking, flatter distribution. And with T → 0: P → [1, 0, 0] — greedy. That's all temperature is: a divisor on logits that sharpens (T<1) or flattens (T>1) the distribution.

Two production-relevant details interviewers love:

  • Numerical stability: e^1000 overflows. Every real implementation subtracts max(z) from all logits first — mathematically identical (it cancels in the ratio), numerically safe. Writing softmax without this is a classic screening-question trap.
  • Why e and not something else? Exponentials make the log-likelihood loss mathematically clean (the gradient of softmax + cross-entropy is beautifully simple: P − y), and they guarantee strict positivity while exaggerating differences — a "soft" version of argmax, hence the name.

1.4 Cross-entropy: measuring wrongness

Cross-entropy loss for next-token prediction is simply: −log(probability the model gave to the correct token). Model said 0.665 for the right answer → loss = −log(0.665) = 0.408. Model said 0.001 → loss = 6.9. Perfect confidence → loss = 0.

Why not mean-squared error? Two reasons that make a strong interview answer: (1) with softmax outputs, cross-entropy yields the clean gradient P − y while MSE's gradient contains a term that vanishes exactly when the model is confidently wrong — the worst time to stop learning; (2) −log punishes confident mistakes brutally (asymptotically infinite), which is the correct incentive for a probability model. When you see "the model achieved a loss of 2.1," that means on average it assigned e^(−2.1) ≈ 12% probability to each correct next token — and perplexity is just e^loss ≈ 8.2, "as confused as choosing among 8 equally likely options."

1.5 Gradients and the chain rule: the learning signal

The gradient of the loss with respect to a parameter answers: "if I nudge this parameter up by a tiny amount, does the loss go up or down, and how fast?" Collect that answer for all parameters and you have a direction — step the opposite way (gradient descent), and the loss decreases.
w ← w − η · ∂L/∂w    (η = learning rate)

The chain rule is how we compute gradients through composed functions: if L depends on y, and y depends on w, then ∂L/∂w = ∂L/∂y · ∂y/∂w. A neural network is a giant composition — so the gradient at any weight is a product of local derivatives along the path from that weight to the loss. Backpropagation is nothing more than computing these products efficiently, back to front, reusing shared subpaths. It is bookkeeping, not magic.

Train a 2-parameter "network" by hand. Model: ŷ = w·x + b. Data point: x = 2, target y = 10. Start: w = 1, b = 0, learning rate η = 0.1. Loss = (ŷ − y)².

Forward: ŷ = 1·2 + 0 = 2. Loss = (2 − 10)² = 64.
Backward (chain rule): ∂L/∂ŷ = 2(ŷ − y) = 2(−8) = −16.
∂ŷ/∂w = x = 2 → ∂L/∂w = −16 × 2 = −32.   ∂ŷ/∂b = 1 → ∂L/∂b = −16.
Update: w = 1 − 0.1(−32) = 4.2.   b = 0 − 0.1(−16) = 1.6.
Forward again: ŷ = 4.2·2 + 1.6 = 10.0. Loss ≈ 0. One step, huge improvement (toy-sized, but the mechanism is identical at 70B parameters).
"Backprop is how the model learns." — Backprop only computes gradients. The optimizer (SGD, Adam) does the learning by applying updates. Adam, used by essentially all LLM training, adapts a per-parameter learning rate using running averages of the gradient (momentum) and its square (scale) — this is why it converges where plain SGD struggles on transformers. Knowing this distinction is a reliable interview differentiator.

Autograd in 40 lines — the heart of PyTorch

TypeScript — the essential mechanism of every deep-learning framework (full version in the lab)

class Value {
  grad = 0;
  private backward_: () => void = () => {};
  constructor(public data: number, private parents: Value[] = []) {}

  add(o: Value): Value {
    const out = new Value(this.data + o.data, [this, o]);
    out.backward_ = () => {          // d(a+b)/da = 1, d(a+b)/db = 1
      this.grad += out.grad;         // += because a node can feed many paths
      o.grad += out.grad;
    };
    return out;
  }
  mul(o: Value): Value {
    const out = new Value(this.data * o.data, [this, o]);
    out.backward_ = () => {          // d(a·b)/da = b  — the chain rule, locally
      this.grad += o.data * out.grad;
      o.grad += this.data * out.grad;
    };
    return out;
  }
  backward() {                      // topological sort, then chain rule in reverse
    const topo: Value[] = []; const seen = new Set<Value>();
    const build = (v: Value) => { if (!seen.has(v)) { seen.add(v);
      v.parents.forEach(build); topo.push(v); } };
    build(this);
    this.grad = 1;                  // dL/dL = 1
    for (const v of topo.reverse()) v.backward_();
  }
}

That += in the backward closures is the chain rule's sum-over-paths; the reversed topological order guarantees every node's gradient is complete before it propagates further back. PyTorch is this idea, industrialized.

Interview ammunition

Why does everyone use cosine similarity for embeddings instead of Euclidean distance?

Embedding direction encodes semantics while magnitude often encodes nuisance factors (frequency, length). Cosine ignores magnitude. Also, on normalized vectors, cosine, dot product, and Euclidean distance produce identical rankings (monotonically related) — so we normalize once and use dot product for speed. Bonus: name that ranking equivalence — it shows real understanding rather than recipe-following.

What breaks if you implement softmax exactly from its formula?

Overflow: e^z explodes for logits above ~700 (float64). Fix: subtract max(z) from all logits first — the result is mathematically unchanged because the factor e^(−max) cancels in numerator and denominator. This is the standard "do they actually write numerical code" screen.

Why cross-entropy rather than MSE for classification/next-token prediction?

Combined with softmax it gives gradient (P − y): large when confidently wrong, never saturating. MSE through a softmax/sigmoid has a derivative factor that goes to zero precisely when the model is confidently wrong, so learning stalls. Also cross-entropy is the maximum-likelihood objective for a categorical distribution — it's not an arbitrary choice.

What's the difference between backpropagation and gradient descent?

Backprop computes ∂L/∂w for every weight via the chain rule (reverse-mode automatic differentiation — one backward pass costs about the same as one forward pass regardless of parameter count, which is why training giant nets is feasible at all). Gradient descent (or Adam) then uses those gradients to update weights. Separate algorithms, often conflated.

A training run's loss is 2.0. What does that number mean, concretely?

Cross-entropy 2.0 nats → the model assigns on average e^(−2.0) ≈ 13.5% probability to the true next token; perplexity e² ≈ 7.4, i.e., "as uncertain as a fair choice among ~7 tokens." Being able to translate loss into perplexity into plain English is a strong signal.
