Chapter 9. Evaluation: stop trusting vibes

~35 min read · Lab: labs/09-eval · What separates demos from products

Anyone can build a RAG demo that looks great on five hand-picked questions. Shipping a reliable system requires measuring quality — systematically, reproducibly, and automatically on every change. This is the skill that most distinguishes senior LLM engineers, and it comes up in nearly every serious interview: "how do you know it works?"

9.1 The RAG triad

TruLens framed RAG quality as three questions, each isolating a different failure point:

MetricQuestionCatches
Context relevanceDid retrieval fetch the right passages?Retrieval failures (chunking, embedding, query)
Groundedness / faithfulnessIs every claim in the answer supported by the retrieved context?Hallucination — the answer went beyond its evidence
Answer relevanceDoes the answer actually address the question?On-topic-but-useless answers
Grading a student's open-book essay: did they open the right pages (context relevance)? did they represent those pages accurately (groundedness)? did they answer the actual question (answer relevance)? A fluent essay built from the wrong pages, or one that invents facts not on the pages it cited, fails — even though it "sounds" right. These three catch different failures, so you need all three.

9.2 Ragas metrics and how they're computed

Ragas operationalizes these with LLM-as-judge. The formulas are worth knowing:

  • Faithfulness = (claims in the answer supported by the context) / (total claims in the answer). The judge decomposes the answer into atomic claims, then checks each against the retrieved context. 0.8 means 1 in 5 claims is unsupported.
  • Answer relevancy ≈ average cosine similarity between the original question and several questions an LLM generates from the answer — if the answer implies your question, it was relevant.
  • Context precision = of the retrieved chunks, how many were actually useful, weighted toward the top ranks (are the good chunks ranked high?).
  • Context recall = of the facts needed to answer (from a reference answer), how many appear in the retrieved context. Low recall = retrieval missed necessary information.
"LLM-as-judge is objective because it's a model." — LLM judges have real, documented biases: they prefer longer answers, favor outputs from their own model family, are sensitive to position/ordering in pairwise comparisons, and can be swayed by confident tone over correctness. Mitigate: use a strong judge model, give it a precise rubric with examples, use pairwise comparison over absolute scoring where possible, randomize positions, and — critically — spot-check judge decisions against human labels on a sample. Never treat a judge score as ground truth without validating the judge itself. Saying this unprompted signals maturity.

9.3 Golden datasets — your source of truth

A golden set is 30–100 question/answer pairs you (or domain experts) verified by hand for a representative slice of your corpus. It's the reference every automated metric is scored against. Build it deliberately:

  • Seed it from real failures (your Chapter 7 failures.json) plus real user questions — not questions you know it can answer.
  • Cover the hard categories: multi-hop, "not in the corpus" (to test abstention), ambiguous, exact-term.
  • Version it in git like code. When it grows, you're building an asset competitors can't easily copy.
  • Synthetic generation (have an LLM draft Q&A from your docs) bootstraps volume, but a human must curate — ungated synthetic sets measure the generator's biases, not your system.

9.4 Evaluation as CI — the mindset shift

The leap from hobbyist to professional: treat prompts, chunk sizes, and retrieval configs like code, and gate changes on evals. Every PR runs the golden set; if faithfulness drops below threshold, the build fails — exactly like a failing unit test. This is what makes iteration safe: you can try HyDE, a new reranker, a different chunk size, and know within minutes whether it helped or regressed, per metric.

promptfoo config — evals that fail CI (conceptual)

# promptfooconfig.yaml
prompts: [file://rag-prompt.txt]
providers: [file://ragPipeline.ts]        # your whole pipeline as the "model"
tests: file://golden-set.yaml              # the 30+ verified Q&A pairs
defaultTest:
  assert:
    - type: context-faithfulness          # LLM-judge: claims supported by context?
      threshold: 0.9                       # build FAILS below this
    - type: answer-relevance
      threshold: 0.8
    - type: latency
      threshold: 3000                      # ms — quality AND speed are gated

9.5 Tracing and observability

Metrics tell you that something regressed; tracing tells you where. Tools like Langfuse or LangSmith record a span for every stage of every query — query rewrite, each retrieval, rerank scores, the final generation — with inputs, outputs, latency, and cost. When an answer is wrong, you open its trace and see whether retrieval missed the chunk (retrieval bug) or had it and the model ignored it (generation bug) — the retrieve-vs-generate split from Chapter 7, made mechanical. In production, tracing plus your eval set becomes a monitoring canary: run the golden set on a schedule against prod and alert on score drops (a model-provider update or a corrupted index shows up immediately).

Interview ammunition

How do you evaluate a RAG system?

Separate retrieval and generation quality. Retrieval: context precision/recall against a golden set (are the needed chunks retrieved and ranked high?). Generation: faithfulness (claims supported by context — the anti-hallucination metric) and answer relevance. Score with LLM-as-judge but validate the judge against human labels. Wrap it as CI so every change is gated, and trace every stage so failures are localizable. The headline: build a versioned golden set from real failures and never ship a change that regresses it.

What's the most important RAG metric and why?

Faithfulness/groundedness, usually — because a confidently wrong, unsupported answer is worse than "I don't know": it destroys user trust and can cause real harm in domains like legal/medical/finance. It directly measures hallucination, the failure mode RAG exists to prevent. That said, faithfulness alone can be gamed by a system that only ever says "not in the docs," so it must be paired with answer relevance and context recall. The right answer names faithfulness AND why it can't stand alone.

What are the pitfalls of LLM-as-judge?

Length bias (prefers verbose answers), self-preference (favors its own model family), position bias in pairwise comparisons, and susceptibility to confident tone over correctness. Mitigations: strong judge + explicit rubric with examples, pairwise over absolute where possible, randomized positions, and spot-checking judge outputs against human labels to measure the judge's own accuracy. Treating the judge as infallible is the mistake.

How would you catch a production regression from a model-provider update?

Run the golden eval set on a schedule against production (a canary), track metric trends over time in your tracing/eval dashboard, and alert on statistically meaningful drops in faithfulness/relevance/latency. Because the eval is automated and versioned, a silent provider change or index corruption surfaces as a metric dip rather than as user complaints. Same philosophy as monitoring + synthetic checks in classic SRE.

Whiteboard summary

RAG triad: context relevance (retrieval), groundedness/faithfulness (no hallucination), answer relevance (on-target).
Ragas: faithfulness = supported claims / all claims; context precision (good chunks ranked high) & recall (needed facts retrieved).
LLM-judge is biased (length, self-preference, position) → validate against humans.
Golden set = 30–100 hand-verified pairs from real failures; versioned in git; synthetic needs curation.
Evals as CI: gate every change; trace every stage → localize retrieve vs generate; golden set on a timer = prod canary.
