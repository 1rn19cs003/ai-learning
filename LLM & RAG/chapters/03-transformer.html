Chapter 3. The transformer, derived

~50 min read · Lab: labs/03-attention · The most important chapter in Part I

In 2017, "Attention Is All You Need" replaced recurrence with a single idea: let every token directly query every other token. This chapter derives that mechanism with actual numbers you can verify by hand, then assembles the full architecture — multi-head attention, MLPs, residual streams, RoPE, the KV cache — and finishes with the memory math that explains why long context windows are expensive.

3.1 The problem attention solves

"The animal didn't cross the street because it was too tired." What does it refer to? Every token's meaning depends on other tokens — sometimes far away. Pre-transformer models (RNNs/LSTMs) passed information left-to-right through a bottleneck of fixed size: by the time you reach "it", the memory of "animal" has been squeezed through many steps and partially lost. Worse, that sequential chain cannot be parallelized on GPUs. Attention fixes both at once: direct pairwise access (any token can look at any token in one step) and full parallelism (all pairs computed simultaneously as matrix multiplies).

3.2 Query, Key, Value — the retrieval metaphor made literal

For each token vector x, three learned projections produce three vectors: Q = WQx ("what am I looking for?"), K = WKx ("what can I be found by?"), V = WVx ("what do I contribute if selected?"). Attention scores are dot products between one token's Q and every token's K; the scores (after softmax) weight a sum of V's.
Attention(Q, K, V) = softmax( QKᵀ / √dk ) V
It is literally a soft database lookup: Q is the search query, K's are the index keys, V's are the stored payloads. Instead of returning the single best match, softmax returns a weighted blend of all matches. Foreshadowing: RAG (Chapter 7) is this same query-key-value pattern implemented outside the model with an actual database — recognizing that symmetry is a genuinely impressive interview moment.

The complete computation, by hand

Three tokens, dk = 2 (absurdly small so we can verify everything). Suppose after projection:
Q₃ (for token 3, "it") = [1, 1]  ·  K₁ = [1, 0], K₂ = [0, −1], K₃ = [1, 1]  ·  V₁ = [2, 0], V₂ = [0, 2], V₃ = [1, 1]

Step 1 — raw scores (Q₃ · Kᵢ): s₁ = 1·1+1·0 = 1 · s₂ = 0−1 = −1 · s₃ = 1+1 = 2
Step 2 — scale by √dk = √2 ≈ 1.414: [0.707, −0.707, 1.414]
Step 3 — softmax: e^0.707 = 2.028, e^−0.707 = 0.493, e^1.414 = 4.113 → sum = 6.634
weights = [0.306, 0.074, 0.620]
Step 4 — weighted sum of values: out = 0.306·[2,0] + 0.074·[0,2] + 0.620·[1,1] ≈ [1.231, 0.769] (using unrounded weights)

Token 3's new representation (weight 0.620 on token 3, 0.306 on token 1) is mostly its own value plus a substantial contribution from token 1 — "it" has absorbed information from "animal." Every attention layer in GPT-4 does exactly this, just with dk=128 and thousands of tokens.

Why divide by √dk? If Q and K components are independent with variance 1, their dot product has variance dk — scores grow with dimension. Large scores push softmax into its saturated region (winner-takes-all), where gradients vanish and training stalls. Dividing by √dk restores variance ≈ 1. It's a numerical-health trick with a one-line justification — know it cold; it is the single most asked transformer question.

Causal masking — no peeking

For next-token prediction, token i may only attend to tokens ≤ i. Implementation: before softmax, set all "future" scores to −∞ (their softmax weight becomes exactly 0). This mask is why the same architecture can train on all positions of a sequence in parallel — each position predicts its next token using only its past — which is the training-efficiency miracle that made LLMs feasible.

3.3 Multi-head attention: parallel perspectives

One attention pattern per layer is too restrictive — "it" needs grammar relationships AND coreference AND semantic context. Solution: split d_model into h independent heads (GPT-3: 96 heads × 128 dims), each with its own WQ, WK, WV, each computing attention in its own subspace; concatenate outputs, project back. Interpretability research (Anthropic's induction-head work) confirms heads specialize: some track syntax, some copy repeated patterns, some attend to positional neighbors. Compute cost is the same as one full-width head — the split is free diversity.

3.4 The full block: attention gathers, MLP thinks

One transformer block (pre-norm variant, as used by GPT-2 onward). A model is N of these stacked: GPT-3 has 96.
  • Residual stream: each sublayer computes a delta and adds it: x = x + Attn(LN(x)), then x = x + MLP(LN(x)). Two payoffs: gradients flow backward through the identity path unimpeded (making 96-layer training possible), and conceptually the stream is a shared workspace each layer reads from and writes to.
  • LayerNorm re-centers and re-scales each vector — volume control preventing activations from drifting; modern models place it before each sublayer (pre-norm) for stability.
  • The MLP (d → 4d → d, with a GELU nonlinearity) is where per-token computation and most factual-knowledge storage happens — roughly two-thirds of total parameters. Attention gathers; the MLP digests.

3.5 Position: why RoPE won

Attention is order-blind — "dog bites man" and "man bites dog" produce identical attention inputs without positional information. The original paper added sinusoidal position vectors to embeddings. Modern models use RoPE (Rotary Position Embedding): instead of adding position, rotate each Q and K vector by an angle proportional to its position (pairs of dimensions rotate at different frequencies). The elegant consequence: the dot product between a rotated Q at position m and rotated K at position n depends only on the offset m − n — relative position falls out of the geometry automatically. Relative encoding generalizes better, and RoPE's frequency-based form is what context-extension tricks (YaRN, NTK scaling) manipulate to stretch trained models to longer windows.

3.6 The KV cache — why generation is fast, and why long contexts cost memory

Generating token 501 requires attention over tokens 1–500 — but their K and V vectors haven't changed since we computed them. So we cache them. Each new token computes only its own Q, K, V, attends against the cache, appends. Generation cost per token becomes linear in context length instead of quadratic.

KV-cache memory for a Llama-2-7B-shaped model (32 layers, 32 heads, head_dim 128, fp16 = 2 bytes):
Per token: 2 (K and V) × 32 layers × 32 heads × 128 dims × 2 bytes = 524,288 bytes ≈ 0.5 MB
At 4,096 context: ≈ 2.1 GB — per sequence, on top of the 14 GB of weights.
At a hypothetical 128k context: ≈ 67 GB — more than the weights themselves!

This single calculation explains: why long-context serving is expensive, why providers charge more for it, why grouped-query attention (multiple Q heads sharing one K/V head — Llama-2-70B uses 8 KV heads for 64 Q heads, an 8× cache reduction) exists, and why prompt caching (Chapter 11) is such a big cost lever. Interviewers love this calculation because it connects architecture to money.

Single-head causal attention in TypeScript

The worked example, generalized (verify it reproduces [1.231, 0.769] in the lab)

function attention(Q: number[][], K: number[][], V: number[][], causal = true): number[][] {
  const dk = Q[0].length, scale = 1 / Math.sqrt(dk);
  return Q.map((q, i) => {
    const scores = K.map((k, j) =>
      causal && j > i ? -Infinity                       // causal mask: no peeking at the future
      : q.reduce((s, qv, d) => s + qv * k[d], 0) * scale);
    const m = Math.max(...scores);                       // numerically stable softmax (Ch 1!)
    const exps = scores.map(s => Math.exp(s - m));
    const sum = exps.reduce((a, b) => a + b, 0);
    const w = exps.map(e => e / sum);                    // attention weights for token i
    return V[0].map((_, d) =>                            // weighted sum of value vectors
      V.reduce((s, v, j) => s + w[j] * v[d], 0));
  });
}

"The model 'reads' my prompt then 'writes' an answer as two different processes." — There is only one operation: predict the next token given everything so far, repeated. "Reading" is one parallel forward pass filling the KV cache (the prefill — this is the latency you feel before the first word appears); "writing" is the same computation done once per generated token against that cache (decode). Prefill is compute-bound, decode is memory-bandwidth-bound — that asymmetry drives most serving-infrastructure design.

Interview ammunition

Walk me through scaled dot-product attention. Why the scaling?

Project each token to Q, K, V. Scores = QKᵀ: every query dotted with every key = pairwise relevance. Divide by √dk: dot products of d-dimensional unit-variance vectors have variance d, and large-magnitude scores saturate softmax → near-zero gradients → dead training. Softmax rows → weights → weighted sum of V. Output: each token re-expressed as a relevance-weighted mixture of what other tokens offer. (Then do the 3-token numeric example from this chapter on the whiteboard — few candidates can.)

Why does attention cost O(n²), and what does the KV cache change?

Every token attends to every other: n² score pairs per layer per head — that's prefill. The KV cache stores each token's K,V after computing them once, so each generated token costs O(n) against the cache instead of recomputing O(n²). Trade: memory. ~0.5 MB/token for a 7B model in fp16, so 128k context ≈ 67 GB — which is why GQA/MQA (sharing KV heads) and paged attention exist.

Why did decoder-only architectures win over encoder-decoder?

One stack, one objective (next-token), every token position provides training signal, natural streaming generation, and the causal mask lets one sequence train all positions in parallel. Encoder-decoder (T5-style) still shines for pure seq-to-seq like translation, but for general-purpose generation the simplicity and scaling behavior of decoder-only dominated. GPT/Llama/Claude are all decoder-only.

What's actually stored in the MLP layers?

Most of the parameters (~2/3) and, per interpretability research, much of the factual-association knowledge — attention routes information between positions, MLPs transform it per position. This is why "the model knows X" degrades gracefully: knowledge is distributed across many MLP weights, not stored in one place. It's also part of why fine-tuning facts in is unreliable versus RAG (Chapter 4 and 7).

How does RoPE allow context extension after training?

RoPE encodes relative position via rotation frequencies. Positions beyond training length produce rotation angles the model never saw — but because the encoding is a smooth function of position, you can rescale frequencies (NTK/YaRN interpolation) so unseen absolute positions map into the seen angular range, then lightly fine-tune. You're compressing "position space" rather than teaching new positions from scratch.

Whiteboard summary

Attn(Q,K,V) = softmax(QKᵀ/√d)V — derive it, don't recite it: scores → scale (variance) → mask (−∞ future) → softmax → blend V's.
Multi-head = same cost, h independent subspaces. Block: x += Attn(LN(x)); x += MLP(LN(x)). Residual stream = shared workspace + gradient highway.
MLP = 4× wide, ~2/3 of params, per-token "thinking" + fact storage.
RoPE: rotate Q,K by position; dot product depends on offset only.
KV cache: 2·layers·heads·dhead·2B per token (~0.5 MB @7B) — memory buys speed; prefill compute-bound, decode bandwidth-bound.
